What strikes me most isn't the speed — it's how we collaborate through the blurred job boundaries.

I bring a working proof-of-concept into client meetings instead of a mockup, make changes during a conversation, and show results before it ends. That shift changes not just how fast design moves — it changes who gets to participate in decisions.

How I Start

First Principles Before Tools

Every engagement starts the same way: understand the constraint before touching the canvas. Not the stated constraint — the structural one. What is the system actually optimizing for, and where is that in tension with what users need? That gap is usually where the real design problem lives.

I spend more time reading — support logs, product reviews, engineering specs, user research — than most designers expect before I open Figma. The canvas is for expressing a position, not forming one.

The most important design decisions are usually made before the first frame is drawn.

Personas + MNML Customer Journey Map — structural research artifacts built before any interface work begins

This matters especially in AI systems, where the temptation is to start with the interface and work backward to the model. The right order is the opposite: understand what the system can reliably know, what it cannot, and how to design around that boundary honestly.

The Workflow

Token Pipeline to Proof of Concept

I work in a fully connected pipeline: Figma Variables → Style Dictionary → Tailwind config → Storybook → CI/CD. Design tokens flow directly into the codebase. When a token changes in Figma, it propagates to every component and every platform without a manual handoff.

This isn't just an efficiency gain. It removes the translation layer where intent gets lost — where a designer says "8px gap" and an engineer interprets "approximately 8px" and a product manager approves something in between. The pipeline makes the design system the single source of truth, not a reference document.

From there, proof-of-concepts are built in code — real components, real interactions, real data where possible. Not to prototype fidelity, but to decision fidelity. The question is always: is this enough to make the next decision? If yes, ship it to the room.

Principles

What I Hold Constant

  • Constraint before canvas Understand the structural problem before opening a design tool. The canvas is for expressing a position, not forming one.
  • Proof over pitch A working artifact that answers the next question is worth more than a polished presentation of a finished answer. Lead with the minimum demonstrable thing.
  • Tokens are vocabulary, not grammar Design systems need more than named values. They need composition rules, pattern decisions, and governance — the grammar that makes vocabulary coherent.
  • Honour what already exists Every product has accumulated meaning — conventions users rely on, patterns that carry expectation. Earn the right to change something before changing it.
  • AI earns trust, it doesn't claim it In AI systems, transparency is not a feature — it's a precondition for use. Intent, confidence, provenance, and alternatives must be present before the interface asks for human judgment.
  • The user role is a decision model Design for what users need to decide, not what they want to see. Different roles require different information architectures from the same underlying data.
  • Ship to learn, not to finish The goal of a release is not completion — it is a question answered by real usage. Define the question before shipping.
